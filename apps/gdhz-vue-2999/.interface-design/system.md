# 广东省海洋灾害综合决策系统设计系统

> 最后更新：2026-03-06
> 适用范围：`apps/gdhz-vue-2999`

---

## 方向与气质

**产品定位：** 面向值班指挥、业务研判和监测联动场景的海洋灾害综合决策界面。

**目标感受：** 明亮、中性、清晰、克制。

**设计原则：**
- 信息先于装饰
- 文字先于氛围
- 强调先于炫技
- 稳定先于炫目

**当前统一方向：** 不再使用“深海科技风”“冰蓝发光控制台”作为默认视觉叙事。系统进入亮色指挥界面阶段，核心判断标准是可读性、确定性和信息效率。

---

## 颜色系统

### 背景层级

| 令牌 | 语义 | 原则 |
|------|------|------|
| `--bg-deepest` | 页面背景 | 明亮、干净、低噪声 |
| `--bg-deep` | 大区块背景 | 用于页面层级过渡 |
| `--bg-panel` | 面板背景 | 轻玻璃感，但必须保证文字清晰 |
| `--bg-card` | 卡片背景 | 比面板更轻、更纯净 |
| `--bg-hover` | 轻交互态 | 仅作微弱反馈 |
| `--bg-active` | 激活态底色 | 比 hover 稍强，但不喧宾夺主 |

### 文字层级

| 令牌 | 用途 | 要求 |
|------|------|------|
| `--text-primary` | 页面主标题、核心数值、关键正文 | 使用中性深墨色，保证高确定性 |
| `--text-secondary` | 正文、说明、主要标签 | 保持稳定、易读，不带明显青蓝色相 |
| `--text-tertiary` | 次级标签、说明文字 | 允许弱化，但必须清晰可辨 |
| `--text-muted` | 时间戳、辅助说明、极弱信息 | 仅用于最低优先级文本 |
| `--text-inverse` | 深色背景上的文字 | 仅在必须反白时使用 |

**强约束：** 正文和高频标签不允许默认使用青灰蓝文字。蓝色只用于交互、选中、链接或特定高亮，不用于大面积正文。

### 强调色边界

| 类别 | 用途 | 规则 |
|------|------|------|
| `--accent-cyan` / `--accent-blue` | 交互焦点、选中态、局部重点 | 只在焦点位置出现，不作为全局默认文字色 |
| `--alert-red` | 高危预警、严重异常 | 必须明显，但只服务风险信息 |
| `--alert-orange` / `--alert-yellow` | 中低等级预警 | 与业务等级绑定，不随意借用 |
| `--alert-blue` | 蓝色预警或业务蓝态 | 不得泛化到普通正文 |
| `--status-online` / `--status-warn` / `--status-offline` | 设备或业务状态 | 必须配合文案，不只靠颜色表达 |

**禁止项：**
- 面板标题默认使用高饱和强调色
- 大面积 cyan glow 作为日常装饰
- 用强调色代替信息层级

## 字体与排版

### 字体角色

| 令牌 | 用途 |
|------|------|
| `--font-body` | 中文正文、标签、标题、说明 |
| `--font-display` | 数值强化、少量强调数字 |

**原则：** 中文界面优先使用稳定、清晰、笔画干净的字体；展示字体只用于数字强化，不作为整套标题的默认人格。

### 字号底线

| 层级 | 推荐 |
|------|------|
| 页面/模块主标题 | 16px-20px |
| 面板标题 | 14px |
| 核心正文 | 13px-14px |
| 次级正文/标签 | 12px |
| 极小角标 | 11px |

**硬性规则：**
- 不再把 10px 作为普遍标签字号
- 11px 只允许用于极少量角标、状态胶囊或空间受限信息
- 中文正文不得低于 12px

### 排版原则

- 标题不依赖大写和字距夸张来制造“科技感”
- 渐变字不是默认方案
- 副文本必须弱于主文本，但不能虚到影响辨识
- 信息优先靠大小、粗细、留白分层，而不是靠颜色堆层级

## 面板规范

### 面板结构

标准结构：
1. `panel-header`
2. `panel-summary`（按需）
3. `panel-content`

### 面板视觉原则

- 面板标题默认使用 `--text-primary`
- 面板强调色只作为辅助身份线索，不直接占据主要阅读入口
- 顶部发光线只能弱化存在，不能成为主视觉
- 面板背景允许轻微玻璃感，但不得降低正文对比度
- hover 反馈以轻微底色变化为主，不使用强 glow

### Badge 与状态

- badge 可以保留业务色
- badge 必须短小明确，不能替代正文层级
- 状态色需要有足够留白，不能把整个面板染色

---

## Header 与导航规范

### Header 原则

- 品牌区以秩序和识别为主，不以装饰插画为主
- 标题用实体字，不用渐变字做默认表现
- 时间、天气等工具信息要稳定、清楚、不过度弱化

### 导航原则

- 导航默认是清晰、安静的容器
- 激活态应该明确，但不需要液态高光、强发光或夸张滤镜
- 选中态优先使用背景层级、边框、阴影和字重差异表达
- 未选中态要清晰可辨，不能因为透明度过低而发灰发虚

---

## 动效与质感

### 动效原则

- 动效必须服务状态变化或信息进入
- 页面默认动效要稀疏、克制、可预期
- 不再以呼吸光、持续发光、波浪高光作为默认氛围组件

### 阴影与边框

- 阴影用于建立层次，不用于制造“悬浮炫光”
- 边框优先用低对比中性边框
- 高光只在小范围焦点区域出现

---

## 禁止项

以下做法默认视为不符合当前设计系统：

- 深海蓝/青蓝作为整套文字主色
- 大面积半透明蓝灰字
- 面板标题统一用强调色
- 默认使用 10px/11px 中文正文
- 渐变字、发光字、液态玻璃高光作为常规标题/导航方案
- 用炫技效果替代信息分层
- 组件直接硬编码旧色值而绕过全局 token

---

## 开发约束

- 新组件优先消费 `variables.css` 中的令牌
- 组件内若新增颜色，必须先判断能否复用现有 token
- 业务状态色允许局部特化，但普通文本不得脱离全局文字体系
- 视觉调整先改 token，再改组件，不允许只在单个组件里打补丁式修色
