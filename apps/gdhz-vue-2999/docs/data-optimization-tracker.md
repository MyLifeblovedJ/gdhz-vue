## 2. 已实施优化

1. 区县数据改为按需加载（避免进入首屏主包）。
2. 区县仅在达到缩放阈值时渲染（2D/3D 分别控制）。
3. 区县仅保留沿海相关范围显示（过滤非沿海区域）。
4. 区县标签去重（避免“中山/中山市”重复显示）。

## 3. 当前待优化项（按优先级）

### P0（优先）

1. 区县大数据渲染进一步减负  
现状：仍可能在首次放大触发时有渲染峰值。  
建议：对区县几何做分级简化（多份 GeoJSON，对应不同缩放层级）。

2. 手型悬停时滚轮偶发卡住问题（已缓解未根治）  
现状：概率已降低，但刷新早期窗口仍偶发。  
追踪：见 `docs/adr/0005-interactive-hover-wheel-stall-investigation.md`。

### P1

1. 预警/台风/设备 Mock 数据后端化  
现状：全部在前端文件内，更新需重新发版。  
建议：迁移到 API，前端只做展示和缓存。

2. 行政边界数据服务化  
建议路线：静态 MVT（优先） -> 再评估是否需要 PostGIS/GeoServer。

### P2

1. 全量 GeoJSON 资源版本化（hash + metadata）  
目标：便于缓存控制、回滚、排障。

2. 数据质量检查脚本  
目标：自动检查字段一致性（`Name/code/geometry`）、重复名称、空几何。

## 4. 推荐目标架构（阶段性）

### 阶段 A（短期，低成本）

1. 保持当前前端懒加载方案；
2. 增加区县数据“分级简化版”；
3. 补充性能埋点（首屏、首次放大、首次区县渲染耗时）。

### 阶段 B（中期，性能提升明显）

1. 将区县/地市边界切为静态矢量瓦片（MVT）；
2. 通过 Nginx/CDN 提供 `pbf`；
3. 前端按地图层级请求瓦片，替代大块 GeoJSON 解析。

### 阶段 C（长期，平台化）

1. 若出现复杂空间查询与在线更新需求，再引入 PostGIS/GeoServer。
