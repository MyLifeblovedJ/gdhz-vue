# AI 对话面板迁移接入说明

本文档给负责“将当前 GDHZ AI 对话面板迁移到另一台服务器/另一个网站”的同事使用。

文档日期：2026-03-08  
当前服务状态以本文档编写当日的在线环境为准。

## 1. 当前可用服务信息

### 1.1 对外访问入口

- 当前服务器内网 IP：`172.30.76.36`
- 当前对外访问入口：`http://172.30.76.36:3000`
- AI 接口基址：`http://172.30.76.36:3000/api/ai`
- 健康检查：`http://172.30.76.36:3000/api/health`

说明：

- 现在外部系统应当统一走 `3000`。
- `3000` 由 `nginx` 对外提供服务，并已显式反代 `/api/` 到本机 BFF。
- 不需要直接访问内部 `3001`。

### 1.2 内部端口说明

- `3000`：对外入口，推荐其他系统访问这个端口
- `3001`：AI BFF 内部服务端口，不建议对外暴露
- `3100`：前端 Vite 开发服务内部端口，不建议对外暴露
- `25808`：AionUi 内部服务端口，不建议对外暴露

结论：

- 迁移接入时，对方系统应优先对接 `http://172.30.76.36:3000/api/ai`
- 不要让浏览器直接请求 `3001`、`3100`、`25808`

## 2. 推荐接入方式

### 2.1 推荐方案

在“另一台服务器”的 `nginx` 上增加一条反向代理，把本系统的 AI 服务映射为本地路径。

推荐映射为：

- 另一网站本地路径：`/gdhz-ai/`
- 上游目标：`http://172.30.76.36:3000/api/ai/`

示例配置：

```nginx
location /gdhz-ai/ {
    proxy_pass http://172.30.76.36:3000/api/ai/;
    proxy_http_version 1.1;
    proxy_read_timeout 3600s;
    proxy_send_timeout 3600s;
    proxy_buffering off;

    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}
```

这样做的好处：

- 前端不用跨域直连远端 `3000`
- 不依赖浏览器 CORS
- SSE 流式响应更稳
- 便于后续切换域名、HTTPS、鉴权

### 2.2 次选方案

如果对方项目不方便加 `nginx` 反代，也可以直接在前端请求：

- `http://172.30.76.36:3000/api/ai`

当前 BFF 允许跨域，但这不是首选方案。优先用反代。

## 3. 前端配置方法

当前 GDHZ 前端的 AI API 封装默认读取：

- `VITE_API_BASE_URL`

在另一个项目里建议这样配置：

- 如果使用本地反代：`VITE_API_BASE_URL=/gdhz-ai`
- 如果直接请求远端：`VITE_API_BASE_URL=http://172.30.76.36:3000/api/ai`

注意：

- 当前 GDHZ 项目里默认是 `/api`
- 如果你不是完整复制当前前端，而是迁移到另一个已有网站，建议改成专用前缀，例如 `/gdhz-ai`
- 这样不会和对方项目自己的 `/api` 冲突

## 4. 请求头要求

所有 AI 请求都应带以下请求头：

- `x-tenant-id: gdhz`
- `x-user-id: <稳定的用户唯一标识>`

建议：

- `x-user-id` 不能每次刷新都变
- 应该基于登录用户 ID，或者至少持久化到 `localStorage`
- 因为历史会话、历史消息、待审批工具都依赖这个值进行归属

如果 `x-user-id` 不稳定，会出现：

- 看不到之前的聊天历史
- 每次都像新用户
- 会话列表和确认操作对不上

## 5. 需要对接的接口

### 5.1 最低可用接口

如果只是做一个“能发问、能收到流式回复”的最小对话框，最少只需要：

- `POST /chat/stream`

如果通过反代接入，完整地址就是：

- `POST /gdhz-ai/chat/stream`

如果直接请求远端，完整地址就是：

- `POST http://172.30.76.36:3000/api/ai/chat/stream`

### 5.2 完整复刻所需接口

要完整复刻当前 3000 页面中的 AI 对话面板，需要接以下接口：

1. `POST /chat/stream`
2. `GET /catalog`
3. `GET /sessions`
4. `GET /history`
5. `GET /confirmations`
6. `POST /confirm`
7. `POST /summary/current`
8. `PATCH /sessions/:chatSessionId`
9. `DELETE /sessions/:chatSessionId`

## 6. 接口定义

### 6.1 流式对话

接口：

- `POST /chat/stream`

请求头：

- `Content-Type: application/json`
- `Accept: text/event-stream`
- `x-tenant-id: gdhz`
- `x-user-id: <稳定用户ID>`

请求体示例：

```json
{
  "chatSessionId": null,
  "message": "请结合当前页面信息给出风险分析",
  "context": {
    "page": "home",
    "region": "gd"
  },
  "selection": {
    "backendKey": "",
    "backend": "",
    "modelId": ""
  }
}
```

说明：

- 首轮对话时，`chatSessionId` 传 `null` 或空字符串
- 后续轮次，继续传服务端返回的 `chatSessionId`
- `selection` 用于指定 Gemini/Codex 和具体模型

### 6.2 AI 目录

接口：

- `GET /catalog`

用途：

- 拉取可用后端列表
- 拉取模型列表
- 显示当前默认后端和默认模型

当前在线环境已验证可返回：

- `gemini`
- `codex`

### 6.3 会话列表

接口：

- `GET /sessions?page=1&pageSize=20`

用途：

- 拉取会话历史列表
- 在左侧会话栏展示

### 6.4 聊天历史

接口：

- `GET /history?chatSessionId=<ID>&page=1&pageSize=200`

用途：

- 切换会话后拉取历史消息
- 恢复完整聊天记录

### 6.5 待审批工具列表

接口：

- `GET /confirmations?chatSessionId=<ID>`

用途：

- 查询当前会话下待确认的工具调用

### 6.6 工具审批

接口：

- `POST /confirm`

请求体示例：

```json
{
  "chatSessionId": "会话ID",
  "callId": "工具调用ID",
  "msgId": "可选，消息ID",
  "data": {
    "decision": "approve"
  }
}
```

说明：

- `callId` 必填
- `chatSessionId` 必填
- `data` 的具体结构由审批选项决定，按服务端返回的待审批内容原样提交即可

### 6.7 当前摘要

接口：

- `POST /summary/current`

用途：

- 对“当前态势摘要类问题”走摘要接口
- 当前 GDHZ 前端会在某些问题类型下优先调用这个接口

请求体示例：

```json
{
  "region": "gd",
  "timeRange": "24h",
  "detailLevel": "standard",
  "snapshot": {},
  "selection": {
    "backendKey": "",
    "backend": "",
    "modelId": ""
  }
}
```

### 6.8 重命名会话

接口：

- `PATCH /sessions/:chatSessionId`

请求体示例：

```json
{
  "title": "新的会话标题"
}
```

### 6.9 删除会话

接口：

- `DELETE /sessions/:chatSessionId`

## 7. SSE 事件协议

`/chat/stream` 返回的是 `text/event-stream`，前端要解析标准 SSE 事件。

当前服务端会发送以下事件：

1. `meta`
2. `delta`
3. `confirm_add`
4. `confirm_update`
5. `confirm_remove`
6. `stream_event`
7. `done`
8. `error`

### 7.1 事件含义

- `meta`：返回本轮会话元信息，通常含 `chatSessionId`、模型选择信息
- `delta`：返回增量文本，前端需要不断拼接
- `confirm_add`：新增一个待审批工具
- `confirm_update`：更新待审批工具状态
- `confirm_remove`：移除一个待审批工具
- `stream_event`：过程事件，用于展示 “View Steps” 或工具执行轨迹
- `done`：本轮流结束，通常携带最终 `reply` 和 `chatSessionId`
- `error`：本轮失败

### 7.2 最低处理要求

如果只是先做一个能用的聊天框，至少要正确处理：

- `delta`
- `done`
- `error`

如果要完整复刻当前面板，还要处理：

- `meta`
- `confirm_add`
- `confirm_update`
- `confirm_remove`
- `stream_event`

## 8. 最小前端流程

### 8.1 首次进入

1. 调 `GET /catalog`，拿到可用后端和模型
2. 初始化一个本地稳定的 `userId`
3. 默认不传 `chatSessionId`

### 8.2 用户发送消息

1. 调 `POST /chat/stream`
2. 监听 `delta`，逐步更新 AI 回复内容
3. 监听 `meta` 或 `done`，拿到 `chatSessionId`
4. 把 `chatSessionId` 保存在当前会话状态中

### 8.3 后续连续对话

1. 继续调用 `POST /chat/stream`
2. 请求体里带上同一个 `chatSessionId`

### 8.4 恢复历史会话

1. 调 `GET /sessions`
2. 用户点击某个会话
3. 调 `GET /history?chatSessionId=...`
4. 把返回消息重新渲染出来
5. 额外调 `GET /confirmations?chatSessionId=...` 恢复待审批项

## 9. curl 联调示例

### 9.1 健康检查

```bash
curl http://172.30.76.36:3000/api/health
```

### 9.2 拉取模型目录

```bash
curl \
  -H 'x-tenant-id: gdhz' \
  -H 'x-user-id: test-user-001' \
  http://172.30.76.36:3000/api/ai/catalog
```

```bash
curl -N \
  -H 'Content-Type: application/json' \
  -H 'Accept: text/event-stream' \
  -H 'x-tenant-id: gdhz' \
  -H 'x-user-id: test-user-001' \
  -X POST \
  -d '{
    "chatSessionId": null,
    "message": "请分析广东当前风险态势",
    "context": {
      "page": "home",
      "region": "gd"
    },
    "selection": {
      "backendKey": "gemini",
      "backend": "gemini",
      "modelId": "auto"
    }
  }' \
  http://172.30.76.36:3000/api/ai/chat/stream
```

## 10. 迁移时建议直接复用的代码

如果要完整复刻当前功能，不建议从零重写协议，建议直接参考或复用以下文件：

- 前端 API 封装：`gdhz-vue/src/api/ai.js`
- 对话面板主逻辑：`gdhz-vue/src/components/map/FloatingToolbar.vue`

建议重点复用：

- SSE 事件解析逻辑
- `x-user-id` 持久化逻辑
- 会话列表/历史消息恢复逻辑
- 工具审批逻辑
- 流式过程步骤展示逻辑

## 11. 对接注意事项

1. 另一项目如果本身也有 `/api`，不要强行复用 `/api`，改成 `/gdhz-ai` 更稳。
2. 必须保证 `x-user-id` 稳定，否则历史会话无法正确显示。
3. SSE 反代必须关闭缓冲：`proxy_buffering off;`
4. 超时时间要足够长，建议至少 `3600s`
5. 不要尝试直接接 `3001`、`3100`、`25808`
6. 如果未来 `172.30.76.36` 发生变化，需要同步更新目标反代地址

## 12. 最终结论

给迁移同事的最简任务说明如下：

1. 在新服务器 `nginx` 上，把 `/gdhz-ai/` 反代到 `http://172.30.76.36:3000/api/ai/`
2. 前端把 AI 接口基址设为 `/gdhz-ai`
3. 所有请求带上 `x-tenant-id: gdhz` 和稳定的 `x-user-id`
4. 最少接通 `POST /chat/stream`
5. 若要完整复刻，再补 `catalog`、`sessions`、`history`、`confirmations`、`confirm`、`summary/current`、会话重命名和删除

按以上方式处理后，另一网站即可完整复刻当前 GDHZ 的 AI 对话能力。
